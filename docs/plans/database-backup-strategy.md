## Problem Statement

シフト管理システム（Next.js 16 + Prisma 6 + PostgreSQL）にバックアップ手段が存在しない。本番運用は自社サーバ／VPS／オンプレ環境を想定しており、PostgreSQL を自前で運用するため、データ消失・ハードウェア故障・操作ミス・ランサムウェアに対する保護を自分たちで設計・実装する必要がある。

## Status Quo

現時点でバックアップに関するドキュメント・スクリプト・運用手順は確認できない。`docs/` 配下にバックアップ関連の資料はなく、`docs/shift_database_schema_v9.md` にもバックアップ章はない。仮に今 DB ボリュームが破損したら復旧手段は存在せず、過去のシフト履歴・変更履歴・労務証跡が全損する。

## 失った時のダメージ評価

| データ種別 | 失った時の痛み | 影響範囲 |
|---|---|---|
| 直近1週間のシフト | 紙やExcelで現場に出ているなら手で再構築できる | 数日の混乱 |
| 過去のシフト履歴・`shift_change_history` 等の履歴テーブル | 「いつ誰がシフトを変えたか」が消える | 監査・労務トラブル時に証明できない |
| 従業員マスタ・グループ・役職 | あまり変わらないので比較的痛くない | 影響小 |
| 認証情報（`User` テーブル） | 全員ログイン不可 | 即時業務停止 |

## 許容レベル（Level 2: 標準）

- **RPO（許容データロス）:** 数分以内
- **RTO（許容復旧時間）:** 数時間以内
- **保持期間:** 直近35日分の PITR 可、月次フルは1年保持
- **整合性検証:** 月次でリストア訓練を実施、四半期で別環境への完全復元

このレベルを選んだ理由: `shift_change_history` 等の履歴テーブルが既にしっかり設計されているため、「履歴の証跡が消えるのは困る」の優先度が高い。履歴を残す設計をしているのに、それごと失われたら設計意図が無駄になる。

## Constraints

- 本番運用環境は自社サーバ／VPS／オンプレ。クラウドのマネージド DB ではない。
- バックアップ転送先はクラウドストレージ不可、**社内 NAS または別の社内サーバ**に転送する。
- `employees` テーブルに従業員氏名・カナ等の個人情報を含む。保存・転送ともに暗号化必須。
- DB 本体は PostgreSQL（バージョンは `prisma/schema.prisma` の datasource を参照、運用環境のバージョン要確認）。

## Premises（合意済み）

1. **バックアップ対象は PostgreSQL DB のみで十分。** アプリコードは git、`.env` は別途管理されているため、DB が戻ればアプリは再構築可能。
2. **個人情報を含むため、保存・転送ともに暗号化必須。**
3. **バックアップ先は必ず DB 本体と物理的に別の場所（社内 NAS or 別サーバ）。** 同一サーバ内に置いてもディスク故障・ランサムで一緒に飛ぶので意味なし。
4. **「バックアップが取れている」と「復元できる」は別物。** 月1回のリストア訓練を運用に組み込む。

## Approaches Considered

### Approach A: 最小構成（cron + pg_dump + 別ホスト転送）

**実装:** `pg_dump` を毎晩 cron で実行 → gzip + `gpg`/`age` で暗号化 → rsync で別サーバ/NAS へ転送 → 古いものは7世代でローテーション。

- 工数: S（CC で 30分、人手 1日）
- リスク: Low
- Pros: シンプル、依存ツールほぼゼロ、誰でも理解できる
- Cons: PITR できない、最大24時間ロスになるため Level 2 要件未達、増分なし
- **判定: Level 2 要件未達のため不採用**

### Approach B: pgBackRest + 社内 NAS（採用）

**実装:** `pgBackRest` で日次フル + 時間別差分 + WAL ストリーミングアーカイブ → 社内 NAS（NFS/CIFS マウント）または別サーバへ rsync over SSH → 月次で別環境にリストアして整合性確認。

- 工数: M（CC で 2〜3時間、人手 3〜5日）
- リスク: Low-Med
- Pros: PostgreSQL 界隈の事実上の標準、増分・並列・暗号化を内蔵、PITR 完全サポート、設定がほぼ宣言的、運用情報・トラブルシュート例が豊富
- Cons: 学習コスト数時間、最初の環境構築が必要

### Approach C: Barman + ストリーミングレプリケーション

**実装:** プライマリ DB から Barman サーバへ WAL をストリーミング → Barman 自身も別 NAS へ rsync。さらに別サーバに read-only レプリカ。

- 工数: L（CC で 1日、人手 1〜2週間）
- リスク: Med
- Pros: ほぼゼロロス、レプリカで読み取り負荷分散も可能
- Cons: Barman サーバ管理が増える、レプリカ運用は別の知識がいる、現規模だと過剰投資

## Recommended Approach: B（pgBackRest + 社内 NAS）

### 構成

```
┌─────────────────────┐                    ┌──────────────────────┐
│  PostgreSQL Primary │  WAL push →        │  Backup Host          │
│  (本番アプリサーバ)  │  (archive_command) │  (pgBackRest)         │
│                     │                    │                       │
│  /var/lib/postgres  │  SSH (連続)        │  /var/lib/pgbackrest  │  ← これが PITR ソース
└─────────────────────┘                    │  (ローカルディスク)   │     (常に最新)
                                            └──────────┬───────────┘
                                                       │
                                                       │ rsync over SSH (時間別 + 日次)
                                                       │ ※ NAS はコールドコピー
                                                       ▼
                                            ┌──────────────────────┐
                                            │  社内 NAS            │
                                            │  /backup/postgres    │
                                            │  35日保持 + 月次1年   │
                                            └──────────┬───────────┘
                                                       │
                                                       │ 月次でこのスナップショットを使い
                                                       ▼
                                            ┌──────────────────────┐
                                            │  リストア検証環境    │
                                            │  (別 VM / 別ホスト)  │
                                            └──────────────────────┘
```

**重要な構成判断:** pgBackRest の repo は**バックアップホストのローカルディスクに置き**、NAS へは rsync で別途同期する。NAS を NFS/CIFS で直接マウントして repo にするのは公式に非推奨（ロック・fsync semantics の問題でリポジトリ破損リスクあり）。NAS 側はコールドコピーで、災害時にバックアップホストごと失った場合の最終リカバリ源として位置づける。PITR の主ソースは常にローカルディスク上の repo。

### スケジュール

| 種別 | 頻度 | 保持期間 |
|---|---|---|
| WAL アーカイブ | 連続（数分単位） | 35日 |
| 差分バックアップ | 平日日次（深夜2時） | 7日 |
| フルバックアップ | 週次（日曜深夜2時） | 5週（=35日） |
| 月次フル（長期保持） | 月初 | 1年 |
| リストア訓練 | 月次 | 結果ログを四半期保管 |

**設計判断:** Level 2 の「35日以内 PITR」を満たすには、フル世代をその全期間カバーできるように残す必要がある。フル日次にすると 35本=DBサイズの35倍の容量が必要になり、社内 NAS の容量を圧迫する。フル週次 + 差分日次なら 5本で35日カバーでき、容量効率が良い。RPO は WAL アーカイブが連続で動いているため、フル頻度に依存せず数分以内を維持できる。

### 暗号化とキー管理

- pgBackRest の `repo1-cipher-type=aes-256-cbc` を有効化
- **重要: 暗号化はリポジトリ初期化（`stanza-create`）時に決定する。後から変更不可で、再初期化が必要になる。** → 実装ステップ3 で必ず先に決定する
- パスフレーズ（`repo1-cipher-pass`）は社内シークレット管理ツール（Vault 等）で管理。`.env` や `pgbackrest.conf` に平文で書かない
- **キーエスクロー必須:** パスフレーズを失うと全バックアップが復号不能になる。バックアップとは**物理的に別の保管場所**（社内金庫、別Vault、紙印刷を複数管理者で分散保管 等）に最低2拠点に保管。年1回ローテーション + 訓練時に復号確認
- NAS への rsync は SSH 経由で転送暗号化

### 実装ステップ

#### 1. バックアップホスト＋プライマリ両方への pgBackRest 導入

**両ホストにインストール必須:**
- バックアップホスト: フル/差分バックアップを実行、repo を保持
- **プライマリ: `archive_command` から呼び出すために pgBackRest クライアントが必要、設定ファイル（pg1-path 等）も配置必要**

両ホストで実行: `apt install pgbackrest` または `yum install pgbackrest`

**SSH 通信は双方向:**
- バックアップホスト → プライマリ: バックアップ取得時（`pgbackrest backup` で pg1-host 経由）
- プライマリ → バックアップホスト: archive_command 実行時（WAL を repo へ push）

両方向で SSH 鍵交換が必要。後述「SSH 鍵交換のセキュリティ姿勢」を双方向に適用すること。

NTP（chronyd / systemd-timesyncd）が両ホストで動作していることを確認。**PITR の精度はクロック同期に依存する。**

#### 2. PostgreSQL 設定（プライマリ側 `postgresql.conf`）

```conf
# WAL 関連
wal_level = replica                  # archive 以上必須（pgBackRest のデフォルト要件）
archive_mode = on
archive_command = 'pgbackrest --stanza=shift archive-push %p'
archive_timeout = 60                 # WAL を最低60秒に1回はクローズ。RPO数分要件のため

# レプリケーション関連（将来 standby 追加余地のため）
max_wal_senders = 3
wal_keep_size = 1GB                  # archive_command 一時失敗時のバッファ

# WAL ディスク圧迫対策（archive_command 連続失敗時に DB 停止を防ぐ）
# pg_wal が満杯になると DB が停止する。ディスク監視と組み合わせる
```

**SSH 鍵交換のセキュリティ姿勢:**
- バックアップホスト → プライマリ への SSH は専用 OS ユーザー (`pgbackrest`) で行い、`/etc/passwd` のシェルは `/usr/bin/pgbackrest` に固定するか、または `~/.ssh/authorized_keys` で `command="pgbackrest"` 強制 + `restrict` オプションを付与してインタラクティブシェルを禁止する
- プライマリ側でこのユーザーには sudo 権限を**与えない**（pgBackRest はプライマリ上の `postgres` 実行権限を必要としない、PostgreSQL の replication ユーザー権限で動く）
- 鍵は ed25519 を採用、3ヶ月ごとローテーション、廃止鍵の `authorized_keys` 削除を runbook に明記
- ログ: プライマリ側で `auth.log` のバックアップユーザーログインを fail2ban もしくは中央ログに集約し、想定外時刻のログインを検知

`pg_hba.conf` にバックアップユーザー用の行を追加し、`replication` ロールを付与した専用ユーザーを作る:

```sql
-- PostgreSQL 15+ の場合
CREATE ROLE pgbackrest WITH REPLICATION LOGIN PASSWORD '...';
GRANT EXECUTE ON FUNCTION pg_backup_start(text, boolean) TO pgbackrest;
GRANT EXECUTE ON FUNCTION pg_backup_stop(boolean) TO pgbackrest;

-- PostgreSQL 14 以前は関数名が異なる:
-- GRANT EXECUTE ON FUNCTION pg_start_backup(text, boolean, boolean) TO pgbackrest;
-- GRANT EXECUTE ON FUNCTION pg_stop_backup(boolean, boolean) TO pgbackrest;
-- 本番 PostgreSQL バージョンを Open Questions で確定後、該当する GRANT を採用すること
```

#### 3. pgBackRest 設定（バックアップホスト側 `/etc/pgbackrest/pgbackrest.conf`）

```ini
[global]
repo1-path=/var/lib/pgbackrest
repo1-retention-full=5               # フル週次×5週 = 35日（Level 2 の PITR 要件と整合）
repo1-retention-diff=7               # 差分保持: 平日日次×1週間 = 最大7本
repo1-retention-archive=full         # フルに紐づく WAL は自動で保持（35日カバー）
repo1-retention-archive-type=full    # 上記と組み合わせて WAL を full の retention に追従させる
repo1-cipher-type=aes-256-cbc
repo1-cipher-pass=<Vaultから注入>     # ファイル直書き禁止、systemd EnvironmentFile 等で渡す
process-max=4
log-level-console=info
log-level-file=detail
start-fast=y
archive-async=y                      # アーカイブ非同期化で archive_command の遅延を吸収

[shift]
pg1-host=<primary-host>
pg1-host-user=postgres
pg1-path=/var/lib/postgresql/<version>/main
pg1-user=pgbackrest
```

**stanza 初期化（一度だけ、暗号化設定確定後に実行）:**
```bash
pgbackrest --stanza=shift stanza-create
pgbackrest --stanza=shift check
```

#### 4. NAS 連携（rsync 方式・マウント直置き禁止）

バックアップホストから NAS への SSH 鍵を交換し、ラッパースクリプト `/usr/local/bin/sync-to-nas.sh` で同期。`--link-dest` で世代差分を作り NAS 容量を節約し、終了時に `latest` シンボリックリンクを今回のスナップショットに張り替える:

```bash
#!/bin/bash
set -euo pipefail
NAS_HOST=nas
NAS_BASE=/backup/postgres
TS=$(date +%Y%m%d-%H%M)
SLACK_WEBHOOK="${SLACK_WEBHOOK_URL:-}"  # systemd EnvironmentFile 等から注入
STAGE="init"

notify_failure() {
  local exit_code=$?
  local msg=":x: sync-to-nas.sh failed at stage=${STAGE} (exit=${exit_code}) on $(hostname) at ${TS}"
  echo "$msg" >&2
  if [ -n "$SLACK_WEBHOOK" ]; then
    curl -fsS -X POST -H 'Content-Type: application/json' \
      -d "$(jq -n --arg t "$msg" '{text:$t}')" "$SLACK_WEBHOOK" || true
  fi
  exit "$exit_code"
}
trap notify_failure ERR

# 前回スナップショットを link-dest にして増分転送
STAGE="link-dest-detect"
ssh "$NAS_HOST" "test -L $NAS_BASE/latest" && LINK_DEST="--link-dest=$NAS_BASE/latest" || LINK_DEST=""

STAGE="rsync"
rsync -az --delete $LINK_DEST /var/lib/pgbackrest/ "$NAS_HOST:$NAS_BASE/$TS/"

# latest シンボリックリンクを今回分に張り替え（NAS 側で）
STAGE="symlink-update"
ssh "$NAS_HOST" "ln -sfn $NAS_BASE/$TS $NAS_BASE/latest"

# 35日より古いスナップショットを削除（月次フルディレクトリは別命名で保護）
STAGE="cleanup"
ssh "$NAS_HOST" "find $NAS_BASE -maxdepth 1 -type d -name '????????-????' -mtime +35 -exec rm -rf {} +"

STAGE="success"
echo "sync-to-nas.sh OK at ${TS}"
```

これで NAS 側にも世代別ディレクトリが残り、`latest` を辿れば直近のコールドコピーに即アクセスできる。

#### 6. cron 登録（バックアップホスト、`postgres` ユーザー）

```cron
# WAL は archive_command で連続プッシュ（cron 不要）

# 差分バックアップ（平日日次・深夜2時、月〜土）
0 2 * * 1-6  pgbackrest --stanza=shift --type=diff backup

# フルバックアップ（週次・日曜深夜2時）
0 2 * * 0    pgbackrest --stanza=shift --type=full backup

# 月初の月次フルは特別命名で長期保持（後述スクリプトで分離）
# 0 3 1 * *   /usr/local/bin/monthly-full.sh

# 整合性検証（日次・深夜4時）
0 4 * * *    pgbackrest --stanza=shift check && pgbackrest --stanza=shift verify

# NAS 同期（バックアップ完了の30分後＝深夜2:30）
30 2 * * *   /usr/local/bin/sync-to-nas.sh

# 月次リストア訓練（毎月1日 朝3時）
0 3 1 * *  /usr/local/bin/restore-drill.sh
```

#### 7. リストア訓練スクリプト（`/usr/local/bin/restore-drill.sh`）

別 VM 上で `pgbackrest restore --type=time --target='YYYY-MM-DD HH:MM:SS'` を実行 → PostgreSQL 起動 → 整合性チェック SQL を流す。

**本番誤実行ガード（必須）:** リストア検証環境を初期セットアップする際に、専用 sentinel テーブルを作成しておく。スクリプトはこの sentinel が存在する DB でしか動作しないようにする。

```sql
-- 検証環境セットアップ時に1度だけ実行（runbook に記載）
CREATE TABLE _restore_drill_marker (
  marker_id INT PRIMARY KEY DEFAULT 1,
  env_name TEXT NOT NULL DEFAULT 'restore-drill',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
INSERT INTO _restore_drill_marker (marker_id) VALUES (1);
```

```bash
# restore-drill.sh の冒頭ガード
psql "$DRILL_CONN" -c "SELECT 1 FROM _restore_drill_marker WHERE marker_id=1" -t -A 2>/dev/null | grep -q '^1$' \
  || { echo "ERROR: _restore_drill_marker not found. This is NOT the drill environment. Aborting." >&2; exit 1; }
```

このガードがあれば、誤って本番接続文字列で起動しても sentinel が無いため即 abort する。リストア後の DB は sentinel テーブルを含まないが、これは pgBackRest restore 後にスクリプトが INSERT するのではなく、**pgbackrest restore でリストアされた本番のデータ**に対して検証を流すフロー。検証スクリプト自体は sentinel が存在する別 DB（`postgres` 等のメタ DB）に対して実行され、検証対象 DB の名前を引数で受け取る形が安全:

```bash
# シェル: メタ DB で sentinel チェック → ターゲット DB で検証実行
META_CONN="host=localhost dbname=postgres user=drill"
TARGET_CONN="host=localhost dbname=shift_restored user=drill"

psql "$META_CONN" -c "SELECT 1 FROM _restore_drill_marker WHERE marker_id=1" -t -A | grep -q '^1$' \
  || { echo "ERROR: not in drill env"; exit 1; }

# ここから TARGET_CONN に対して検証を実行
psql "$TARGET_CONN" <<'SQL'
-- 1. 行数の妥当性
SELECT 'employees' as tbl, COUNT(*) FROM employees
UNION ALL SELECT 'shifts', COUNT(*) FROM shifts
UNION ALL SELECT 'shift_change_history', COUNT(*) FROM shift_change_history;

-- 2. 最新シフト日付（直近の business activity が含まれているか）
SELECT MAX(shift_date) FROM shifts;
SELECT MAX(changed_at) FROM shift_change_history;

-- 3. 履歴トリガが復元後も動作することの確認（書き込み伴うが、検証環境のみで実行）
BEGIN;
INSERT INTO shifts (employee_id, shift_date, shift_code) VALUES (...);
SELECT COUNT(*) FROM shift_change_history WHERE changed_at > NOW() - INTERVAL '1 minute';
ROLLBACK;  -- 検証環境でも痕跡を残さない
SQL
```

結果を `/var/log/restore-drill/YYYY-MM.log` に保存し、Slack へサマリ通知。

## 容量見積もり

```
必要容量 = (DB実サイズ × フル世代数) + (差分平均 × 差分世代数) + (WAL量 × 保持日数)

例: DB 5GB, 1日変更率 5%, WAL 1日 500MB のシステム
  - フル: 5GB × 5本 = 25GB
  - 差分: 250MB × 14本 = 3.5GB
  - WAL:  500MB × 35日 = 17.5GB
  - 月次フル: 5GB × 12本 = 60GB
  ----------------------------------
  合計: 約 106GB（圧縮で実効60〜70GB）
```

実装前に本番 DB の実サイズ（`SELECT pg_size_pretty(pg_database_size('shift_db'))`）と1日あたりの WAL 生成量を計測して数値を確定する。

## Open Questions

- 本番運用予定のサーバスペック・PostgreSQL **メジャーバージョン**・ディスク容量は？（pgBackRest はバージョン互換性があり、メジャーバージョンが変わるとリストア手順が変わる）
- 本番 DB の現在サイズと、1日あたりの WAL 生成量は？（容量見積式の DBサイズ・WAL量に必要）
- 社内 NAS の容量と空き状況、書き込み権限、SSH 接続可否は？
- バックアップホストは新規調達か既存サーバ流用か？（pgBackRest は別ホスト推奨）
- Slack/メール通知の社内ルートは確立済みか？
- Vault または同等のシークレット管理ツールは社内に存在するか？無ければ最低限の代替手段を決める必要あり
- **Prisma migrations と PITR の関係:** マイグレーションを跨いだ過去時点に PITR で戻す場合、アプリ側のスキーマ期待値とずれる可能性。リカバリ手順書に「PITR 後はその時点のアプリバージョンを git でチェックアウトしてからデプロイ」を明記する必要

- **「まだ決めていない／一緒に考えたい」と素直に言えた** — 多くの人は知らないことを認めず、雑な要件で発注して後で困る。許容レベルを決めずに設計に進むのが一番危険なので、ここで止めたのは正しい判断だった。
