# アプリレベル暗号化 設計ドキュメント

**作成日**: 2026-06-11
**ステータス**: 設計（未実装）
**対象**: 従業員シフト管理システム（Next.js 16 + Prisma + PostgreSQL）
**意思決定経緯**: `/office-hours` セッション（脅威モデル確定 → 鍵管理トリレンマ → 方式B選択）

---

## 1. 目的とスコープ

サーバー（OS/DBエンジン）の設定に依存せず、**アプリケーション層でPII列を暗号化**する。DBのデータファイルやバックアップが at-rest で読み取られても、暗号鍵がなければ平文を復元できない状態にする。

### 守る脅威（決定済み）
- **脅威B**: OS/DB管理者がデータファイルを直接読む（at-rest）。
- app-level 暗号化はDBが平文を一度も保持しないため、**「ファイル直接読み取り」と「DBに接続してSELECTする内部者」の両方**をカバーする（TDE/BitLockerは前者のみ）。
- 副次効果として**脅威A（持ち出されたバックアップ/ダンプ）**も自動的にカバー（ダンプ内の対象列が暗号文になる）。

### スコープ外（別課題として明記）
- アプリの**未認証ページ閲覧**（誰でも全ページ閲覧可の設計）はこの暗号化では守れない。DB暗号化＝データ保護完了ではない。
- 転送時暗号化（TLS）。
- ログインパスワード（既に bcrypt 済み）。

## 2. 鍵管理: 方式B（起動時パスフレーズ / メモリのみ）

### トリレンマと選択理由
「① OS管理者を防ぐ ② 追加インフラ不要 ③ 無人再起動」は同時に2つまで。脅威B + 外部KMSなしの環境では、**③（無人再起動）を諦めて手動アンロックを受け入れる**のが方式B。

### Envelope（DEK/KEK）方式
パスフレーズから直接データを暗号化せず、二段構えにする。

```
パスフレーズ ──scrypt(+salt)──▶ KEK ──unwrap──▶ DEK（32バイト乱数）──▶ 各PII列を暗号化
                                          ▲
                            disk上に「wrapped DEK + salt」を保存（暗号文・鍵なしでは無価値）
```

- **DEK（Data Encryption Key）**: 32バイト乱数。実際の列暗号化に使う。**メモリ上のみ**で保持し、disk には絶対に書かない。
- **KEK（Key Encryption Key）**: パスフレーズ + salt から **scrypt**（決定。Node 標準 `crypto.scrypt`・新規依存ゼロ）で導出。パラメータは N=2^17 以上を目安（運用者が稀に1回アンロックする用途なので高コストでも許容）。
- **wrapped DEK**: DEK を KEK で暗号化したもの。salt とともに disk に保存（例: `secrets/keyring.json`）。これは暗号文なので、ファイルを読まれても**パスフレーズがなければ復元不可**＝脅威Bを満たす。
- 利点: **パスフレーズ変更 = wrapped DEK の再ラップのみ**（データ再暗号化不要）。DEKローテーション時のみデータ再暗号化（バージョン管理で対応）。

### 暗号方式
- **AES-256-GCM**（認証付き暗号）。Node 標準 `node:crypto` を使用（**新規依存ゼロ** — bcrypt 同様ネイティブで完結）。
- 列フォーマット: バージョンプレフィックス付き単一文字列。
  ```
  v1:<base64(iv)>.<base64(authTag)>.<base64(ciphertext)>
  ```
  - IV: 値ごとにランダム12バイト。
  - `v1:` プレフィックスで将来の鍵/方式ローテーションに対応。

### メモリ保持の限界（正直に明記）
Node のヒープはスワップロックされないため、**稼働中マシンで root/Administrator がプロセスメモリをダンプすれば DEK を抽出され得る**。app-level メモリのみ方式は「at-rest ファイルに対しては強い」が「ライブRAMフォレンジックに対しては完全ではない」。脅威B（ファイル読み取り）には十分。

## 3. Sealed → Unlock の仕組み（nssm 対応）

### 状態遷移
1. nssm が**無人でサービス起動**。アプリは **sealed 状態**（DEK 未ロード）で立ち上がる。
2. 運用者が **localhost 限定のアンロック操作**でパスフレーズを投入。
3. アプリが KEK 導出 → wrapped DEK を unwrap → **DEK をメモリ保持** → `ready` に遷移。

パスフレーズは「運用者の入力 → プロセスメモリ」だけを通り、**disk を経由しない**。

### コンポーネント（設計のみ・未実装）
| 要素 | 役割 |
|---|---|
| `lib/crypto/keyring.ts` | `let dek: Buffer \| null`、`isUnlocked()`、`encrypt()/decrypt()`、`unlock(passphrase)`、`lock()` を提供するモジュール内シングルトン |
| `lib/crypto/envelope.ts` | scrypt KDF、DEKラップ/アンラップ、wrapped DEK ファイルの読み書き |
| `app/api/admin/unlock/route.ts` | POST。**loopback(127.0.0.1)限定** + 試行レート制限。パスフレーズ受領→アンロック |
| `app/api/admin/lock-status/route.ts` | GET。sealed/ready を返す（監視・UIバナー用） |
| `scripts/unlock.ts` | **唯一のアンロック入口（CLIのみ）**。サーバー上でパスフレーズをプロンプト入力し loopback にPOST。Web管理ページは作らない |
| 全体ゲート | sealed 中は PII を含むページで「🔒 ロック中 — 管理者のアンロックが必要です」を表示 |

> アンロック入口は **CLI のみ**（決定）。`app/api/admin/unlock` は CLI と稼働中サーバープロセス間の **IPC チャネル**（別プロセスの CLI からメモリ上の DEK をセットするため）であり、loopback 限定。ユーザー向けWeb管理画面は持たない。

### Next.js 前提（重要な制約）
- `next start` は**単一の長命 Node プロセス**。モジュールレベルの `dek` はプロセス生存中はリクエストをまたいで保持される。nssm 単一プロセス運用と整合。
- **マルチインスタンス/ワーカー構成にすると各プロセスを個別にアンロックする必要**がある（要・将来検討）。dev モードやサーバーレスはこの方式と非互換。

### Sealed 中の挙動
- 暗号化列の**読み取りは復号不可**（名前等が表示できない）。**書き込みも失敗**。
- → sealed 中はアプリが実質「PII について読み書き不可」。明示的なバナー + ステータスで運用者に知らせる。

## 4. 暗号化対象の範囲（段階スコープ）

最大の難所は「最も機微な列＝氏名が、まさに検索・ソート・照合に使われている」点。列ごとに区別する。

**決定スコープ: Tier1 + Tier3（氏名/フリガナ + 自由記述）。Tier2（日付）は初期スコープ外。**

### Tier 1（コア・推奨）: 個人を特定するPII
- `employees.name`、`employees.name_kana`
- **クエリ影響**: `ORDER BY name`（shifts/duty エクスポート）・`LIKE` 検索（employeeSearch）が**DBでは不可**になる。
  - 対策: **従業員数が小規模（数十〜数百）なので、全件取得→復号→アプリ内でソート/フィルタ**が現実的。ページングもアプリ層へ移す。
  - Excel変換の氏名照合は**既にアプリ内で全件比較**しているため、復号を挟むだけで対応可。

### Tier 2: 日付PII
- `employees.hire_date`、`employees.termination_date`
- **クエリ影響**: `terminationDate: null`（在籍者フィルタ）が複数箇所（shift-conversion 等）にある。
  - 対策: **在籍判定用の平文ブール列（例 `is_terminated`）を別途持ち、日付の値だけを暗号化**。null性（在籍/退職）は漏れるが日付値は秘匿。スキーマ追加の手間あり。
- **決定: 初期スコープ外（後回し）**。監査要件で日付暗号化が明示された場合のみ追加する。

### Tier 3（安価・先行実施推奨）: 自由記述
- shift `note`、dutyAssignment `title`/`note`、dutyType `defaultTitle`/`defaultNote`
- **クエリ影響なし**（検索/ソート対象でない）。暗号化が最も安全に入る。Prisma拡張の動作検証に最適。

### スコープ外（推奨: 暗号化しない）
- 組織構造名: `group.name`(@unique)、`functionRole.roleCode`、`position.name`
- 理由: **個人データではない** + `@unique`制約・JOIN・表示が多くクエリコストが最大。暗号化の費用対効果が悪い。

## 5. Prisma 統合

- **Prisma Client Extension（`$extends`）**で対象モデル列の暗号化（write時）/復号（read時）を透過化し、呼び出し側のコードを極力変えない。
  - `query` 拡張: `create/update/upsert` の `data.name` 等を書き込み前に暗号化。
  - `result` 拡張: 読み取り結果の `name` 等を復号。
- **拡張で解決できない部分（要リファクタ）**: `where`/`orderBy` は変換できない。
  - `lib/db/employees.ts` の検索: `where: { name: { contains } }` 不可 → **全件取得→復号→JSフィルタ**（小Nなので可）。将来Nが増えたら**ブラインドインデックス列** `name_bidx = HMAC(dek, normalize(name))`（等価検索用）を検討。
  - エクスポートのソート: 取得後にJSで復号値ソート。

---

## 6. 履歴トリガーとの関係

- DB トリガーは旧/新の列値を履歴表へ**コピーするだけ**。暗号文を保存しても整合は取れ、同じ DEK で復号可能。**トリガー本体の変更は不要**（要確認: トリガー関数が値をパース/比較していないこと）。
- 履歴の**表示コード**も復号拡張を通す必要がある。

## 7. 移行・ロールアウト計画（段階）

| フェーズ | 内容 | リスク |
|---|---|---|
| **P0** | 暗号コア（`lib/crypto/` envelope・keyring・unlock API・CLI・sealedゲート）。スキーマ変更なしでアンロック動線を実証 | 低 |
| **P1** | Tier 3（自由記述notes）を暗号化。Prisma拡張をエンドツーエンドで検証（クエリ影響ゼロ） | 低 |
| **P2** | Tier 1（氏名/フリガナ）。検索/ソートをアプリ層へリファクタ + 既存行の**バックフィル暗号化**（アンロック状態で1回実行） | 中 |
| **P3**（任意） | Tier 2（日付）。在籍フラグ列を併設して暗号化 | 中 |

各フェーズは単体でリリース可能・テスト付き。バックフィルはバージョン付き暗号文（`v1:`）で出力。

## 8. 運用ランブック（必須）

- **再起動・クラッシュ復旧のたびに手動アンロック**が必要（Windows Update 月次再起動含む）。「再起動後はアンロックを実行する」手順書を整備。
- nssm 自動再起動後も sealed で戻る → 人がアンロックするまでダウン継続（24/7必須でない社内アプリ前提で許容）。
- **監視**: sealed 状態がX分以上続いたらアラート（`/api/admin/lock-status`）。
- **バックアップ対象に追加**: `secrets/keyring.json`（wrapped DEK + salt）も必ずバックアップ。これが無いと復元したDBをアンロックできない。

## 9. リスク・トレードオフ（正直な明記）

1. 🔴 **パスフレーズ紛失 = 暗号化列の永久消失**。サーバー側に鍵が無いのが本方式の本質ゆえ、パスフレーズを失えば誰も復号できない。**鍵エスクロー（回復手順）が前提条件**。
2. 🟡 **再起動ごとのダウンタイム**（手動アンロックまで PII 読み書き不可）。
3. 🟡 **検索/ソートのリファクタ**（employees/shifts/duty の db 層）。
4. 🟡 **ライブRAMダンプには非完全**（at-rest ファイルには有効）。
5. 🟡 **マルチインスタンス化で各プロセス個別アンロックが必要**。
6. ⚪ **未認証ページ閲覧は別問題として残る**。

---

## 10. 鍵エスクロー設計（着手前に必須）

メモリのみ鍵は「バス係数1」を作る。回避策:
- 回復用パスフレーズで**第2の wrapped DEK** を作り、オフラインの安全な場所（金庫/組織のシークレット管理）に保管。
- 通常運用パスフレーズとは別に、データ所有者（人事/管理責任者）が回復手段を保持。
- **この回復手順を文書化し、データ所有者の承認を得るまで暗号コードを書かない。**

## 11. 決定事項（2026-06-11 確定）

| 項目 | 決定 | 備考 |
|---|---|---|
| 暗号化スコープ | **Tier1（氏名/フリガナ）+ Tier3（自由記述notes）** | Tier2（日付）は初期スコープ外 |
| KDF | **scrypt（Node標準・依存ゼロ）** | N=2^17 以上。argon2id は不採用（ネイティブ依存回避） |
| アンロックUI | **CLI のみ（`scripts/unlock.ts`）** | Web管理ページは作らない。`/api/admin/unlock` は CLI↔サーバー間のloopback IPC |
| 鍵管理方式 | **方式B（起動時パスフレーズ・メモリのみDEK）** | nssm は sealed→unlock 型で対応 |

- 脅威の精緻化: 防ぎたい「OS/DB管理者」が**ファイル読み取りのみ**か**DBにsuperuser接続もできる**か（**どちらでも方式Bは有効**なので設計に影響なし。確認のみ）。
- **鍵エスクロー（回復）手順の文書化と承認**（下記 The Assignment）— これは実装着手の前提条件。

## The Assignment（次の現実アクション）

コードより先に **鍵エスクロー（回復）手順を1枚にまとめ、データ所有者（人事/管理責任者）に承認をもらう**こと。

理由: 方式Bはサーバーに鍵を残さないのが価値だが、その裏返しで**パスフレーズを失えば全PIIが永久に復元不能**になる。「誰が回復パスフレーズを、どこに、どう保管するか」を決めて承認を得ることが、実装着手の前提条件。これが無いまま暗号化すると、いつか「鍵を失って人事データを丸ごと失う」事故になり得る。
